# File close and resource cleanup

## Problem

Procfs files with dynamic content (e.g., `/proc/meminfo`) must generate a
fresh snapshot each time they are opened. The generated bytes need to live
long enough for the caller to read them, then be freed when the file is
closed. Today there is no close path — `File` has no `Drop` impl and
`FileOps` has no `close`/`release` method, so any per-open allocation must
be `Box::leak`'d and is never reclaimed.

Static procfs entries (like `/proc/version`) sidestep this by storing their
data in the binary, but that approach does not work for content that changes
at runtime.

## What needs to change

1. Add `fn close(&self, file: &mut File)` to `FileOps` with an empty
   default implementation.

2. Implement `Drop` for `File` so that it calls `self.fops.close(self)`
   when the file handle goes out of scope. This mirrors Linux's `release`
   callback in `struct file_operations`.

3. For procfs dynamic entries, `open` would allocate a buffer (or a
   `ProcfsDataInode`) and `close` would free it. The simplest mechanism:
   store a `*mut u8` + length in the `File` (or in a per-filesystem
   wrapper) and deallocate in `close` via
   `alloc::alloc::dealloc` or by reconstructing and dropping a `Box`.

## Scope of impact

- `File` — add `Drop` impl
- `FileOps` — add `close` default method
- `ProcfsFileOps` — implement `close` to free the generated buffer
- Callers that move `File` values need to ensure they are dropped
  (most already do this naturally when the `File` goes out of scope)

## Current workaround

For now, only static procfs entries are supported. Dynamic entries should
not be added until the close path is implemented.
